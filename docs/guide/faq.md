# 常见问题

## 刷入后抓包仍证书错误 / 断网？

先确认模块侧正常：

1. 已**重启**，状态页为「运行正常」
2. 日志里 PID 1、Zygote 注入无报错
3. **强制停止并重新打开**被抓包的 App，以及 Reqable / ProxyPin（已在运行的进程不会自动吃到后续挂载）
4. 若显示「证书集合未生成」或「证书注入失败」，查看日志中的实时源数量与注入结果

若以上都正常，仍出现 TLS 失败或抓包断网，优先怀疑**证书本身不一致**（见下一节），而不是注入又坏了。若只有「单应用白名单」断网、「抓全部」正常，见 [白名单单应用断网](#whitelist-disconnect)。

## 为什么状态正常，抓包还是断网？

**很常见：你本机抓包软件正在使用的根证书，和模块里启用的那张不是同一张。**

### 证书为什么会不一样？

1. **Reqable 与 ProxyPin 不一样**  
   - **Reqable**：安装或「重新生成证书」后，往往会换一对新的根密钥和 CA，系统里的 subject hash 文件名也会跟着变。模块内置只是一份**开箱样例**，**经常不等于**你这台手机 App 里当前正在用的那张。  
   - **ProxyPin**：根证书通常是软件自带的固定 CA，模块内置一般可以直接用；若你自行替换过 ProxyPin 证书，仍需按下方方式导入本机当前证书。  
   名字都叫 Reqable / ProxyPin 也没用——**指纹（公钥）对不上就不算同一张证**。

2. **「用户证书」和「系统证书」不是同一层**  
   Android 把信任库大致分成：
   - **用户证书**（用户凭据 / `cacerts-added`）：设置里手动安装的 CA，很多 App（尤其面向系统信任库的）**不会**按系统 CA 同等信任；
   - **系统证书**（system / Conscrypt APEX 信任库）：本模块注入的目标。抓包要让绝大多数 App 放行，通常必须进**系统**这一层。  
   所以：只在「用户」里装好了、或只在 App 里显示「已安装」，都不等于系统信任库里已经是**你当前那张** CA。

3. **编码和后缀不是一回事**  
   同一张逻辑证书，内容编码常见两种：
   - **PEM**：文本，常见 `-----BEGIN CERTIFICATE-----`，中间是 Base64；
   - **DER**：二进制，文件管理器里往往看不出内容。  
   从抓包软件或系统里导出到「用户证书」时，文件后缀常见 **`.pem`、`.crt`**（也有 `.cer` / `.der`）。**后缀只是习惯命名**：`.crt` 里经常仍是 PEM 文本，也可能是 DER，不能单靠后缀判断。本模块 WebUI / 热挂载按**文件内容**识别，一般不必先改后缀。  
   系统信任库里的文件名则一般是 **8 位十六进制** subject hash（如 `a1b2c3d4.0`），由证书主题计算而来：CA 内容一变，hash 就会变（Reqable 常见）；ProxyPin 若始终用同一张固定 CA，hash 也会保持不变。这不是软件的产品编号。导入后模块会自动算 hash 命名。

4. **文件大小一样，内容仍可能完全不同**  
   两张不同的 CA 完全可能体积接近甚至相同（例如都是同类算法、相近字段长度）。**不能**靠文件名、大小或「都叫 Reqable」判断是否一致，请以 App 里显示的**指纹 / 序列号 / 主题**为准，和导入后的证书对照。

模块内置对 **Reqable** 尤其不能保证等于你 App 里当前那一张；**ProxyPin** 在未自行换证时通常可直接用内置。系统里即便能看到同名条目，指纹不对时，被抓包的 App 仍会 TLS 失败，表现就是断网。

### 怎么处理？

任选其一即可：

1. **推荐 · WebUI 自定义证书（永久，写入系统信任库）**  
   在抓包软件中导出**当前**根证书（常见 `.pem` / `.crt`，内容多为 PEM 或 DER）→ 打开本模块 WebUI「证书」页上传。需要长期使用时，可关掉对应内置开关，只保留你刚导入的自定义证书，避免两张并存造成困惑。保存后**重启**生效。
2. **临时 · 免重启热挂载**（安装时已启用该组件时）  
   把导出的证书放到用户凭据区，或放入存储卡目录（默认 `/sdcard/CertBridge`），再在 WebUI / Action 里选择挂载用户区、存储卡或全部——模块会把它们**提升进系统信任库（临时）**。适合先验证「是不是证书不对」；重启后临时会话会失效，永久方案请仍用自定义导入。
3. **进阶 · 直接放入模块目录**  
   将证书放到模块的 `certs/custom/`，下次开机一并合并进系统信任库（仍建议用 WebUI，便于校验与管理）。

导出后请确认 App 内展示的指纹与导入结果一致，再重试抓包。

其它仍可能导致「看起来已安装、实际断网」的情况：

1. **被抓包的 App 还是旧进程**：Reqable 自己检测通过，不代表微信、浏览器等已更新命名空间。请强停目标 App 后再开，或再重启一次。
2. **注入显示与读库不同步（少见）**：个别机型上，路径能列出证书但进程读库异常。可再重启一次，并查看日志是否有注入相关报错。
3. **抓包软件开了单应用白名单**：白名单为空（抓全部）正常，只勾一个 App 却断网——见下一节，这通常**不是**证书桥没注入成功。

## 白名单为空能抓、只抓单个 App 却断网？ {#whitelist-disconnect}

这是抓包软件（Reqable / ProxyPin 等）的**拦截范围**问题，多数情况下与证书桥无关。

### 现象

- 白名单**为空 / 抓全部 App**：目标 App 能上网，也能抓到包  
- 白名单**只勾某一个 App**：该 App 立刻「没网」，抓包列表里也很少甚至没有请求  

若「抓全部」时已经能解密 TLS，说明系统信任库里的 CA 对这个 App 基本生效；换成单应用白名单后断网，优先查路由 / DNS，而不是反复重装证书。

### 为什么会这样？

抓包工具常用本机 VPN（或等价隧道）做中间人：

| 模式 | 实际行为 |
| --- | --- |
| 白名单为空（抓全部） | 系统 VPN 接管大部分 App，DNS / 路由往往按「全局隧道」处理，路径更完整 |
| 白名单只勾一个 App | 用按应用 VPN：只有该 UID 进隧道，其它 App 直连 |

单应用进隧道后：

1. **没有直连兜底**：该 App 的流量被强制进本地代理；代理、转发、DNS 任一失败，表现就是断网，而不是「抓不到但还能上网」。  
2. **DNS 最容易翻车（很常见）**：只有这个 App 的 DNS 走 VPN；系统 Private DNS / DoH、IPv6 DNS 在单应用模式下更容易解析失败。全局抓包时 DNS 一并进隧道，反而更稳。  
3. **IPv6 / 双栈**：部分实现只接管 IPv4，App 优先走 IPv6 时会直接连不上。  
4. **多进程没进白名单**：主包勾了，`:push`、WebView、独立进程、分身/工作资料未勾，会半通或直接失败；抓全部时这些进程往往也被罩进去。

### 怎么处理？

按顺序试：

1. **先用「抓全部」确认证书**：白名单留空，确认同 App 能抓且不断网。若此时也断网，回到上一节查证书是否一致。  
2. **关掉 Private DNS**：系统设置 → 网络 → 专用 DNS → 关闭，再开单应用白名单试一次。  
3. **在抓包软件里禁用 IPv6 / 仅 IPv4**（若有该选项）。  
4. **白名单勾全相关包名**：主包、分身、同应用多进程入口尽量都勾上。  
5. **判断卡在哪一步**：断网时看抓包列表——若完全没有该 App 的 CONNECT / TLS，更像 DNS 或没进隧道；若有请求但握手失败，再查证书、证书固定（pinning）或双向校验。

**一句话：** 白名单单应用 = 只把目标 App 丢进隧道且失败即断网；抓全部 = 隧道（尤其 DNS）更完整。所以常见是「拦截路径更脆」，不是「白名单把证书弄坏了」。

## 设置里看不到系统证书？

部分系统对「可信凭据 → 系统」展示有延迟。可先看 WebUI 状态，或重启后再查。

## 与其它系统 CA 模块一起用可以吗？

不建议。证书桥会把开机时已可见的其它模块证书一并复制进本次证书集，但后执行的模块仍可能覆盖先前挂载，无法保证加载顺序，请尽量只保留一个。

## WebUI 打不开 / 无法执行命令？

先确认安装时选择了 WebUI，并检查模块目录是否存在 `webroot/index.html`。已安装 WebUI 时，请用 **KernelSU 管理器** 或其它支持模块 WebUI 的客户端打开；普通浏览器没有 `ksu.exec` 桥接。

## 为什么没有“临时免重启挂载”？

自定义安装时若跳过免重启热挂载，模块不会保留 `bin/hot_mount.sh`，已安装的 WebUI 也会隐藏这一功能，模块 Action 的实用菜单也会提示组件未安装。需要使用时可重新刷入同版本，在自定义安装中按音量上启用，或直接选择默认完整安装。

已安装热挂载时，可在模块管理器打开 **Action**：音量上刷新状态，音量下进入实用功能，逐项选择挂载用户区 / 存储卡（默认 `/sdcard/CertBridge`）/ 全部，或卸载当前临时会话。

## 自定义证书无法上传？

模块要求设备提供 OpenSSL，并拒绝无效、过期、非 `CA:TRUE` 或超过 64 KiB 的证书。文件名无需手动改为 hash。

## 为什么修改开关后没有立即生效？

内置开关和 WebUI 自定义证书属于永久配置。为避免重写正在使用的开机证书层，保存后必须重启。需要免重启时，可使用「临时免重启挂载」读取用户凭据区或存储卡证书。

## 临时证书能完全无痕卸载吗？

正常情况下可以。模块为每次临时会话写入唯一标记，卸载时只对仍显示该标记的命名空间执行一次 `umount`，然后删除临时证书目录；系统文件、用户证书和永久配置均不改变。如果其它脚本在其上继续叠加挂载，模块会拒绝误拆并提示重试或重启。

## 为什么部分已经运行的 App 没有立即信任证书？

模块会尝试覆盖 PID 1、Zygote、抓包 App 与当前可访问的活动挂载命名空间，但部分厂商系统会限制访问个别进程命名空间。WebUI 会显示失败数量；可先强制停止并重新打开目标 App，仍无效时卸载临时会话后重试。

## 为什么提示缺少 nsenter 或 OpenSSL？

临时免重启功能需要 `nsenter`、`stat`、`readlink` 以及支持 `subject_hash_old` 的 OpenSSL。部分 Android 7–8 ROM 或精简 Root 环境不提供这些命令，模块会拒绝热挂载而不是降低校验强度；开机永久注入和 WebUI 其它功能不受影响。

## 为什么不再保存系统 CA 基线？

系统更新、APEX 更新和其它模块都可能让旧基线过期。当前版本在每次重启时直接读取尚未被本模块挂载的实时信任库，生成完整且不可变的本次启动证书集。实时源或生成结果不安全时不会执行挂载，因此无需依赖长期保存的副本。
